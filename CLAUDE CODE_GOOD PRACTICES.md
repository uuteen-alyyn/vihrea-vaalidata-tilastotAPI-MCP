# Good Practices — Claude Code Projects

Practices learned from building the FI Election Data MCP and other projects. Apply these at the start of every session and every new project.

---

## Required Project Files

Every project must maintain these four files. Check all of them at the start of every session.

### Implementation_plan.md

Step-by-step plan created **before coding starts**. Split into phases; each phase has goals and acceptance tests.

- Use `[ ]` / `[x]` checkboxes for individual tasks
- **Mark `[ ]` → `[x]` immediately when a task is done.** Do not batch at the end.
- If a phase is complete, also update its heading and Status line
- If the plan changes mid-session, update it before continuing — don't let it go stale

### Logbook.md

Append-only activity log. **Write the logbook entry before anything else when completing a phase or significant work unit.** Never remove existing entries — only add new ones at the bottom.

Entry format:
```
## ENTRY DESCRIPTIVE TITLE YYYY-MM-DD HH:MM:SS
[what was done]
[decisions made]
[test results]
[notes]
```

Include in every entry:
- Files changed
- Build status (clean / errors)
- Test count and pass/fail (e.g. "159/159 passed")
- Commit hash if committed

### BACKLOG.md

Persistent work queue that survives across sessions.

- **At the start of every session, read BACKLOG.md and surface any outstanding items** to the user
- Add items whenever the user requests work that won't be done immediately
- Remove items **only when the user explicitly says the task is done or dropped** — do not remove on your own judgment
- Format: group by priority (🔴 critical / 🟡 medium / 🟢 low) with short status notes and dates

### CLAUDE.md

Project-specific instructions for the coding agent, checked into the repository. Keep it current. If a new practice or constraint emerges during a session, add it here so future sessions inherit it.

## Git & GitHub Workflow

**Commit after every completed phase or significant self-contained change.** Do not accumulate multiple phases in one commit.

### Commit checklist (always run before committing)

1. `npm run build` (or equivalent) — must exit 0
2. `npm test` — all tests must pass; record count in logbook
3. Stage specific files by name — **never `git add .` or `git add -A`** (risks committing `.env`, cache files, secrets)
4. Write a concise commit message: `Phase N: <what changed>` or `fix: <what and why>`

### Push cadence

- Push to GitHub (`git push`) after every commit, or at minimum at the end of every working session
- The remote is the source of truth. Never let local diverge from remote by more than one session
- Always confirm push completed — don't assume it worked

### Never commit

- `.env` files or any file containing tokens or credentials
- Runtime cache files (e.g. `cache-store.json`)
- `node_modules/`
- Binary build artifacts that are regenerated by the build

## Auditing Before Shipping

Before declaring a feature set production-ready, perform systematic audits across the codebase. Document findings in an `audits/` directory with a dated filename (e.g. `TOOL_AUDIT_2026-03-21.md`).

### Severity levels

- 🔴 **Critical** — causes tool failure or silent wrong results. Fix before anything else.
- 🟡 **Medium** — causes confusion, likely wrong behavior, recoverable. Fix in the same session.
- 🟢 **Minor** — inconsistency or description gap, no failure. Fix when convenient.

### What to audit

- **Parameter naming consistency** — asymmetric or ambiguous names cause wrong calls (e.g. `year1`/`year2` vs `baseline_year`/`comparison_year`)
- **Required vs optional** — is a parameter truly required? Can a sensible default be inferred?
- **Silent failures** — does the tool return empty results instead of an error when given bad inputs?
- **Orphaned code** — loaders, helpers, or utilities that exist but are not reachable from any tool
- **Output key stability** — dynamic output keys (constructed at runtime) break callers; use stable names
- **Error message quality** — does every error tell the caller what went wrong and how to recover?

## Designing Tools for LLM Consumption

When building MCP tools or any API consumed by an LLM, apply these patterns consistently.

### Parameter types

- **Always use `z.coerce.number()`** (not `z.number()`) for numeric parameters. LLMs frequently pass numbers as strings (e.g. `"2023"` instead of `2023`). Coercion silently handles this.
- Use descriptive, unambiguous parameter names. Avoid generic names like `year1`/`year2` — use `baseline_year`/`comparison_year` to remove ordering ambiguity.
- Make parameters optional with sensible defaults wherever possible. Fewer required parameters = fewer validation failures from the LLM.

### Error responses

Every error response should include two things:
1. What went wrong (specific, not generic)
2. How to recover — a concrete next step or a resource to read

```typescript
// Good
return errResult(
  `No candidate data for parliamentary 2018.`,
  'Read election://coverage for available candidate years.'
);

// Too generic
return errResult('Invalid year.');
```

### Output field stability

- Output keys must be predictable from the schema. **Never construct output keys at runtime** (e.g. `` `n_${areaLvl}s_scored` `` is bad — the LLM cannot know the key without running the tool).
- For binary-direction outputs (e.g. `direction='strongholds'` vs `direction='weak_zones'`), **always return both keys** and populate only the relevant one. Dynamic keys make output unparseable without knowing what you passed.

### Tool descriptions

- **Disambiguate similar tools**: if two tools overlap, say explicitly in each description when to use that one vs the other.
- **Don't mistake votes for demographics**: tools that rank by vote-share patterns are not demographic tools. Say so.
- **Document constraints in descriptions, not just docs**: if omitting a parameter triggers an expensive fan-out, say so in the parameter description with a WARNING label.

### Discoverability resources

Provide machine-readable resources (MCP Resources, OpenAPI endpoints, etc.) that let the LLM check data availability before making tool calls. Examples:
- `election://coverage` — what data exists
- `election://unit-keys` — valid geographic keys by election type
- `election://metrics` — metric definitions

Reference these in error paths: `hint: 'Read election://coverage for available years.'`

## Testing Standards

- The build tool (`tsc`, `npm run build`, etc.) must pass before any commit
- All tests must pass before any commit
- Record test count in every logbook entry
- Unit tests cover the deterministic computation layer (metrics, normalizers, query engine)
- Integration/live testing is a separate phase — document findings in the logbook

---

## Session Start Checklist

At the start of every coding session:

1. Read `BACKLOG.md` — surface any outstanding items to the user
2. Read `Implementation_plan.md` — know where you are in the plan
3. Check recent `Logbook.md` entries — understand what changed last session
4. Confirm build is clean before adding new code
